Two levers did the work: **(1) an index** (sql/060 trigram GIN) where an ILIKE was
scanning, and **(2) a query rewrite** where a `count(*) OVER()` window was
materializing the whole corpus to return one page. Measure-first (NYS-114)
confirmed each before it shipped.

## What shipped

### T1 — Measurement (the culprits, named)
- **/rates Services initial load** = `count(*) OVER()` in `listRateRows`. The
  window materialized **all 425,687** unpivoted service rows (`rate_table_child_mv`
  × 5 codes) and sorted them just to `LIMIT 50`. Full plan: Seq Scan (129,490) →
  LATERAL unpivot → WindowAgg over 425k → Sort → Limit. Text search was *faster*
  than the no-filter load because the filter shrank the windowed set first.
- **/orgs search** = 150,499-row seq-scan of `org_tin_rosters` + hash join +
  HashAggregate on every keystroke (the ILIKE on `tin_registry.business_name` was
  a small seq-scan; the roster aggregate was the cost).
- **/directory search** = 5-column trigram `BitmapOr`; the `profession`/
  `subspecialty`/`taxonomy` trigram scans cost ~400 ms even when the term is a
  surname that matches none of them.
- **employers / plans** = 8.5 ms on a 3,476-row table. No action.

### T2 — Index program (`sql/060_search_trgm_indexes.sql`, applied live)
- `idx_rtc_display_trgm` — GIN trigram on `rate_table_child_mv(display_name)`.
- `idx_tin_registry_name_trgm` — GIN trigram on `tin_registry(business_name)`.
  This one flipped the /orgs planner from a 150k-row hash join to a **nested loop**
  from the ~1.4k matched TINs into `org_tin_rosters` via `idx_otr_key` — no org
  summary matview needed.
- **NYS-88 safe**: both are non-unique GIN indexes on a matview whose unique key
  (`idx_rate_table_child_grain`) is already plain-column, so `REFRESH …
  CONCURRENTLY` maintains them automatically. **No new step in the post-ingest
  chain** — the indexes ride the existing REFRESH. (Applied with `CREATE INDEX
  CONCURRENTLY`; `ANALYZE`d after.)
- **Repo rewrite** (`lib/repos/rate-rows.ts` `listRateRows`): dropped
  `count(*) OVER()` for **two parallel queries** (page via top-N + incremental
  sort on the grain index; count beside it) — directory's established pattern.
  The count is **skipped entirely past page 0**, so infinite scroll never
  re-pays it. **Count parity verified**: 425,687 = 425,687 unfiltered; 2,497 for
  `smith`. End-to-end through the live app: `total: 425,687` returned, load-more
  correctly returns `total: 0`.

### T3 — Hybrid search UX (the "look faster than they are" part)
- **/rates ServicesPanel** (`components/rates/services-panel.tsx`): now two layers.
  (1) **Instant client-side reduction** — every keystroke filters the already-
  loaded 500 rows with zero round-trip (the `~/Code/sports` feel). (2) **Debounced
  150 ms** (was 250) indexed server query for the full corpus, sliding in under
  the instant preview. **Keep-previous**: rows never blank; an empty client subset
  shows "Searching all published rates…" (not "no results", because the match may
  live beyond the loaded page); the footer shows "N shown" vs "Searching all…".
- **/directory** (`app/(app)/directory/directory-client.tsx`): debounce 250→150 ms
  and an **idempotent client-side pre-filter** in `sortedProviders` (mirrors the
  server ILIKE on name/city/profession/subspecialty). Surgical — the 911-line
  component already kept-previous and was trigram-indexed; this adds the instant
  layer without touching its network/rate maps, pinning, or infinite scroll.

### T4a — ⌘K command palette + unified search API
- **`app/api/search/route.ts`**: one endpoint fanning out (parallel) across the
  existing indexed repo searches — clients, providers, orgs, plans/employers —
  capped at 6 each. Reuses the domain repos; no new search logic. Practitioner-
  gated; PHI note in-file (keystrokes are not `logEvent`-ed — the audit boundary
  is opening a record, which logs itself). Verified live: `smith`/`health`/`aetna`/
  `casey` all return correctly-linked groups (e.g. `casey` → client *Casey Morgan*
  → `/clients/<uuid>`).
- **`components/search/command-palette.tsx`**: global ⌘K / Ctrl+K launcher,
  composed from the `SearchInput` primitive + a portal (mirrors Modal's scrim/
  escape/backdrop — there is no `Command` primitive to reuse). Keyboard nav
  (↑/↓/↵/esc), grouped entity results + static "Go to" destinations, debounced
  150 ms, keep-previous. Mounted in `AppShell` (workspace variant only).
  **Primitive-gap note for ui-agent** below.

## Flags / decisions for the lead

1. **Directory 5-column search residual (23 ms vs 586 ms).** A name-only query is
   23 ms; the 5-column OR pays ~400 ms for profession/subspecialty/taxonomy
   trigram scans even on a surname. Reordering (name-first, broaden only on a thin
   result) or a single combined `search_text` trigram column would fix it, but
   both change user-visible behavior on a **shared surface** — not changed
   unilaterally. Filed as an issue; the hybrid-UX layer masks it meanwhile.
2. **Task 4b (public /search page) DEFERRED.** It's a substantial new public
   marketing surface; I chose to ship the verified core rather than rush it.
   Recommended approach: reuse the `/api/search` shape but **public-scoped —
   providers/orgs/plans only, never clients (PHI)** — behind a public route with
   the marketing `Nav`/`PageHero`. Next-tranche.
3. **Primitive-gap (→ ui-agent).** The command palette is a new *feature*
   component (not a `ui/*` primitive). If the pattern recurs, a `Command`/
   `CommandPalette` primitive is worth promoting; `SearchInput` also doesn't
   forward a ref (I used `autoFocus` instead — a `forwardRef` would let a palette
   focus imperatively). Both are ui-agent's call.
4. **No org summary matview built** — the trigram index alone made /orgs search
   9× faster and it wasn't needed. The /orgs *default* list (no query) is still a
   155 ms full roster aggregate; acceptable with skeletons, but a per-TIN summary
   matview (31,233 rows) would make it instant if the founder wants it. Flagged,
   not built (would add a post-ingest step — cross-seam).
