## Part 1 — why the bell never rang

**It was never broken. It was starved, and then the only rows it ever had were consumed by a robot.**

The first action was to query the table before touching anything. It held **9 rows, all belonging to the founder's
`user_id` (`…001001`, brendan@liminal.demo, admin), and every one already read** — `unread = 0`. The read path is
correct end to end: `GET /api/notifications` signed in as the founder returns all 9 with the right shape.

Two things explain the silence.

**1. Every one of those 9 rows carries the identical `read_at` of `2026-07-19 03:31:29.059352+00`** — a single
mark-all-read. The bell's design is "opening the menu IS the read event". A UI session was working inside
`components/shell/topbar-bell.tsx` that day (`docs/reports/2026-07-19-ui-workspace.md`), so a headless verification
drive almost certainly opened the bell and marked the only rows that ever existed as read, at 03:31, before the
founder ever looked at it.

**2. Nothing autonomous has ever written a row.** Which brings us to the five writers.

### The five writers — which fired, which are dead, and two that are not writers

| # | writer | ever fired? | why |
| --- | --- | --- | --- |
| 1 | `scripts/seed-workspace-notifications.mjs` | **Yes** — all 9 rows | A demo seeder, not a producer. Kinds `event`/`report` only. |
| 2 | `ops/harvest/runner.mjs` | **No** — but not broken | Correctly silent. |
| 3 | `app/api/cron/daily/route.ts` | **No** — and structurally could not have | See below. |
| 4 | `app/api/stripe/webhook/route.ts` | **Not a writer at all** | Premise correction. |
| 5 | `app/api/notifications/route.ts` | **Not a writer at all** | Premise correction. |

**The runner (2)** only writes when a job fails, which is right. Its bell block was added `2026-07-18 21:49` (`ee102dc`)
— *after* the only runner failure on record (`harvest:fhir-status`, `2026-07-17 23:17`). Every night since has been
green. A writer that has never fired because nothing has gone wrong is working, not dead.

**The daily cron route (3)** is the real finding. It has exactly one failure on record, and the row explains itself:

> `daily` · error · `2026-07-18 15:14` · `duration_ms 300000` · *"Killed at the 300s Vercel maxDuration cap mid-chain."*

The process was SIGKILLed inside the refresh loop. `closeRun()` and the `notifyAdmins()` call both sit **downstream of
the kill point**, so they never executed. **The alerting path only fires on failures fast enough to report themselves —
a timeout, the most likely production failure, kills the notifier along with the job.** The route is also demoted to
manual-only (NYS-130), so it is not scheduled at all. Flagged, not fixed: the route is outside this seam and the
structural fix is an ops decision.

**Two premise corrections.** The brief lists five writers; there are three, and only one is autonomous.

- `app/api/stripe/webhook/route.ts` contains **zero** notification code. It imports `sendDisputeAlert`,
  `sendPaymentReceipt` and `sendTherapistPaid` from `lib/email/stripe-notifications` — an **email** module whose
  filename is, I suspect, the whole source of the belief. No payment event has ever been able to reach the bell.

- `app/api/notifications/route.ts` is the **reader**: `GET` returns list + unread, `POST` marks all read.

### What was fixed, and the proof

The bell fetched **once, on mount, and never again**. These tabs stay open for hours and every producer we have writes
while the app is already open (the runner lands ~05:07). A row would sit in the table until someone happened to
reload. It now polls every 60s while visible, and refreshes on focus and visibilitychange (`7543c43`).

Driven against the live database, signed in headless as the founder:

- inserted one row → `GET` reported **`unread: 1`**, `readAt: null`, correct title at the top

- `POST /api/notifications` → **`unread: 0`**

- probe row deleted → table back to its exact baseline of **9 rows, 0 unread**

**The durable fix is Part 3**: `/monitor` is the first autonomous producer this bell has ever had.

## Part 2 — `ops/night-report.mjs`

A plain script, no LLM, no tokens. It writes one `lead_reports` row per day from five sources already on disk and in
the database: `git log` (commits, files, churn bucketed by area), `docs/reports/<date>-*.md` (**the day's agent work,
which nothing else captured**), `sql/` migrations added, `sync_runs`, and corpus row counts.

### The backfill

| date | derived from | growth |
| --- | --- | --- |
| **2026-07-19** | 32 commits · 83 files · +12,393/−1,881 · **9 agent reports** · 1 migration (`sql/053`) · 4 sync runs | not measured — see below |
| **2026-07-20** | 84 commits · **16 agent reports** · 3 migrations (`sql/061`, `062`, `063`) · 5 sync runs | 11 corpus tables counted, baseline stored |

The founder's `2026-07-17` and `2026-07-18` rows were **not touched** (same lengths, same `updated_at`).

**Growth is measured, never asserted.** Counts are diffed against a baseline stashed in the previous report. A
backfill of an older date does **not** present today's counts as that day's — it says so plainly and stores no
baseline, because storing one would poison every future delta. That is why 07-19 shows no growth table and 07-20 does.

### Annotations survive — the rule, and how it is enforced

An existing row is left alone, full stop. `--force` replaces only a row the script wrote that nobody has touched
since, proven by a sha256 embedded in the body. All four cases verified against the live table:

- untouched machine row + `--force` → **replaced**

- no `--force` → **skipped**

- annotation **appended at the end** + `--force` → **refused**

- edit in the **middle of the body** + `--force` → **refused**

That third case is worth calling out because **the first implementation failed it.** The digest originally covered only
the prose above the footer comment, so an annotation appended at the bottom — the most natural place to add one — left
the checksum intact and `--force` destroyed it. The digest now covers the entire stored text with the sha field blanked
to a placeholder. Caught by testing the rule rather than trusting it.

### Live proof, unplanned

The launchd runner fired at 01:04 **minutes after the jobs.json hunk was committed**, picked up the new job, ran it
first in 5s, and correctly **skipped** the existing 07-20 row:

```
[2026-07-21T05:04:05.271Z] night-report: attempt 1/1
[night-report] 84 commits · 16 reports · 3 migrations · 5 sync runs · 11 tables counted
[night-report] SKIPPED — a row for 2026-07-20 already exists. Annotations are never overwritten.
[2026-07-21T05:04:10.127Z] night-report: ok in 5s
```

### Two premise corrections

- **`jobs.json` entries run BEFORE the matview rebuild, not after.** The brief asks for the job to run after the loads
  and rebuild "so the row reflects a settled database"; the runner's job loop finishes before `runDailyRebuild()`, so
  that is not achievable through jobs.json. It also does not need to be — every corpus table is a **base** table,
  untouched by `REFRESH`. The job is placed **first** instead, deliberately: counting before tonight's loads begin is
  what makes the number belong to the day being reported. Running it last would bill tonight's rows to yesterday.

- **A noon-to-noon "night of D" window was tried and abandoned.** The runner fires at 01:04, so that window is still
  *open* when the job runs and the night's own rows are mid-flight. One calendar-day window for every section is
  complete by then, and no section of the report means a different span of time than the one above it.

## Part 3 — `/monitor`

A new primary tab (System group in the sidebar, `route-title.ts` → "Monitor"). Server reads only, `requireRole("admin")`
— verified: practitioner **403**, anonymous **401**.

Six checks and one informational reading, four grid-of-squares meters, rows **grouped by status** rather than listed flat, and a detail panel per
check giving what it measures, its threshold, what to do about it, and the view the number came from. Thresholds live
in **one exported `THRESHOLDS` object**, documented, so tuning the page is a single edit.

### Live now — needs nothing but `DATABASE_URL`

Connection count vs cap · database size and per-table size · unused indexes · dead-tuple
bloat · longest-running queries · transaction age / freeze horizon · matview staleness vs nightly cadence · the
`sync_runs` ledger. Buffer cache hit ratio is read too, but as a reading with no threshold — see below.

**Matview freshness has no catalog source** — Postgres does not record when a `REFRESH` last ran. It is derived from
our own ledger: the newest error-free step naming a view in `sync_runs.steps` is its last good rebuild. The page says
so in the table footer.

**No PHI can reach this page.** The one field that could carry a value from a `WHERE` clause is a running query's SQL;
its string literals are masked **inside the query itself**, so the value never leaves Postgres — not to a log, not to
a browser.

### Gated on a key that does not exist

Compute hours, storage, branch state and autosuspend come from Neon's **control plane**, which a connection string
cannot reach. The `neon`-prefixed variables in `.env.local` and Vercel are connection strings from the integration, not
API access. That panel renders an honest "not configured" state with the instruction inline. **No compute or cost
number is fabricated anywhere on the page.**

**For the founder — the exact steps:**

1. Go to **console.neon.tech → Account settings → API keys → Create new API key**. Name it something like
   `liminal-monitor`. Copy the key immediately; Neon shows it once.

2. Add `NEON_API_KEY=<key>` to **`.env.local`** for local development.

3. Add the same `NEON_API_KEY` in **Vercel → the Liminal project → Settings → Environment Variables**, for all
   environments, then redeploy.

The panel switches itself on when `process.env.NEON_API_KEY` is present; nothing else needs changing.

### The bell's first real producer

`raiseMonitorAlerts()` turns threshold crossings into notifications. **Every alert states the measured value and the
threshold it crossed** — never a bare "something is wrong". Repeats are suppressed for 20h so a standing warning
cannot turn the bell into wallpaper.

Proven end to end: a crossing wrote two rows and the bell read **`unread: 2`**, e.g.

> *Needs attention: Buffer cache hit ratio — 74.2%*
> `Threshold: warn below 85%, fail below 70%. 74.2% of block reads were served from memory…`

A second `POST` wrote nothing and reported both as `suppressed`. Those rows were then **deleted**; the table is back to
9 rows / 0 unread.

### Four premise corrections, all the same lesson

A check that cries on healthy conditions is a check nobody reads. Each of these was found by measuring, not assuming:

- **Cache hit ratio is no longer a check at all — see "Corrected in review" below.** My first pass merely re-tuned
  the threshold to 85/70, which was not enough: at a measured 74% it still fired "Needs attention" on a healthy
  database while its own remediation told the reader to disregard it.

- **Job silence needed per-job cadence.** A single global threshold reported **9 of 15 jobs unhealthy on a green
  night** — most of them `once` MRF manifest jobs that ran a single time, succeeded, and are finished. Silence is now
  judged against each job's own observed gap between its last two runs; a single-run job is a completed one-off, and
  a run in flight is in flight. Now correctly reports 0 unhealthy.

- **Nightly `REFRESH`/`VACUUM` no longer counts toward the long-query verdict.** It is slow by design, every night —
  a live `REFRESH MATERIALIZED VIEW CONCURRENTLY org_tin_rosters` at 76s raised an alert during testing. Maintenance
  is still shown in the panel, labelled, but never decides the status.

- **`rate_service_rows_mv` showing "no record" is correct, not a bug.** It is in the nightly plan (`sync-plan.mjs`),
  added by `sql/063` yesterday — *after* the last completed `daily` run, so the ledger genuinely has no step for it
  yet. The honest empty state did its job instead of inventing an age.

### Corrected in review — the cache hit ratio is a reading, not a check

**The advisor caught a real defect and the fix is committed (`e4a51cc`).** My re-tuned 85/70 threshold still fired
"Needs attention" on a healthy database, and the remediation text I wrote for it said, in effect, *ignore this*. **An
alert you are instructed to ignore is worse than no alert** — it teaches the founder that this page cries wolf, which
is precisely the failure `ALERT_REPEAT_HOURS` exists to prevent. I had applied that same reasoning to jobs and to
maintenance queries and then shipped a check that violated it.

I verified the three facts rather than accepting them, and the third is worse than reported:

- `shared_buffers` is **456 MB against a 23 GB database** — under 2% resident. A low buffer-pool ratio is the
  architecture, not a regression.

- `pg_stat_database.stats_reset` is **NULL** — never reset. The figure is a lifetime average dominated by one-time
  bulk-load and matview-rebuild sequential reads.

- **`neon_stat_file_cache` is not merely uninstalled — it is absent from `pg_available_extensions` entirely.** Neon's
  local file cache sits below `shared_buffers` and absorbs most of what `pg_stat_database` counts as a miss, and on
  this project it cannot be observed at all.

**On windowing:** I took the instruction to sample-and-diff seriously and concluded it cannot be done cleanly here,
which is the escape hatch the advisor allowed. Windowing fixes the second fact only. It cannot fix the first or the
third — a windowed figure still measures a 456 MB pool and is still blind to the layer that actually absorbs the
misses. There is no threshold that means "going to disk" while the denominator is unobservable. Producing one would
also need sample persistence I do not have cleanly in this seam: a new table is a migration in the ops seam, and
module-level in-memory sampling yields an instance-local window varying from seconds to days.

So it is **demoted to an informational reading**: its own "Readings — measured, not judged" card, no threshold, no
status, and structurally unable to alert, because alerts derive from `checks` and a stat is not a check. The meter
stays but renders in neutral teal (`share`), never amber — colouring it by health would re-assert the judgment just
removed. The full context travels with the number **on the page**, not in a code comment, including the fact that
installing `neon_stat_file_cache` is what would earn it a real check. `THRESHOLDS` now carries an explicit note not
to add one back without it.

Verified after the change: `POST /api/monitor` reports **6 checks, 0 crossing, nothing written**; the bell stayed at
its baseline 9 rows / 0 unread; the header reads **Needs attention 0**; and the meter renders 74 teal squares with
zero amber.
