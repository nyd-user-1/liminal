## What shipped

### `sql/063_rate_service_rows.sql` — `rate_service_rows_mv` (new, additive)

One row per `(insurer, billing ID, NPI, network, setting, billing code)` over all 20 codes.
**Long, not pivoted** — `listRateRows` already unpivoted `sql/032` back to this grain with a
`CROSS JOIN LATERAL (VALUES …)`, so the matview now *is* the grain and the unpivot is gone.

Additive rather than a widened `sql/032`, because four other consumers read that matview
(`lib/repos/rate-table.ts` → /published-rates, `lib/repos/analytics.ts`,
`lib/analytics/metrics.ts`, `lib/repos/admin.ts`). Widening it would have meant 40 rate
columns and a DROP+CREATE of a 70 MB shared object. `sql/032` is untouched.

**Built:** 1,091,544 rows, 698 MB, 34 s. `REFRESH … CONCURRENTLY` verified working, 26 s,
non-blocking (unique index is on plain columns — the NYS-88 trap avoided deliberately).
Registered in `ops/harvest/sync-plan.mjs` so it refreshes nightly.

#### The cap moved 100 → 400, and that was a correctness call

`sql/032` excludes a `(billing ID, insurer)` group holding >100 leaves, keeping the
platform-scale TINs (which /orgs owns) out of an inline table. Adding fifteen codes **adds
leaves** — a clinician priced only for 90832 is a new `(npi, network, setting)` leaf — so
re-applying `<=100` to the 20-code leaf set would have *evicted rows /rates lists today*:

| cap | groups | service rows | groups lost vs today | **rows lost vs today** |
|---:|---:|---:|---:|---:|
| 100 | 38,688 | 156,812 | 157 | **9,616** |
| 150 | 38,872 | 179,430 | 62 | 3,977 |
| 200 | 38,959 | 194,315 | 25 | 1,960 |
| 400 | 39,067 | 223,218 | 0 | **0** |

400 is the smallest round threshold that loses nothing. Deleting 9,616 rows a user can see
today, inside a change whose entire purpose is to show *more*, would be a regression wearing
a feature's clothes. It still excludes the 81 platform groups the cap exists for — the
largest is `npi:1235600834` (Empire) at 47,182 leaves, then `npi:1427671437` at 12,088.

**No-loss proof** (`EXCEPT` against the live 5-code surface):


Every one of the 452,168 service rows visible before this change is still present.

## Measurements — before / after

Endpoint timings on :3010 are **not usable evidence today**: eight agents share that dev
server and the same request varied 0.27–1.22 s. So the before/after below is measured at the
Neon driver, running the old and new query shapes back to back in one process (best of 5).

| leg | BEFORE (sql/032 unpivot, 5 codes) | AFTER (sql/063 long, 20 codes) |
|---|---:|---:|
| **first-paint wall (page ∥ count ∥ facets)** | **145 ms** | **54 ms** |
| **search wall (page ∥ count)** | **102 ms** | **36 ms** |
| first paint: page | 54 ms | 22 ms |
| first paint: count | 59 ms | 51 ms |
| first paint: facets | 41 ms | 21 ms |
| search `q=smith`: page | 97 ms | 37 ms |
| search `q=smith`: count | 80 ms | 23 ms |
| code filter (90837): page | 106 ms | 24 ms |
| code filter (90837): count | 50 ms | 28 ms |

Every leg is faster while carrying 2.41× the rows and 4× the codes. Two precomputes paid for
the extra volume — both found by measuring, not by guessing:

**1. The search was seq-scanning.** The Services search is an `OR` across five columns, and
a `gin_trgm` index on `display_name` alone cannot serve that `OR`. `EXPLAIN` showed a
Parallel Seq Scan, 503 ms for the count leg over 1.09M rows. Folding the five into one
pre-lowered `search_text` column with a single trigram index made it a Bitmap Index Scan:

```
count(*) WHERE search_text LIKE '%smith%'   503 ms  ->  3.3 ms   (5,978 rows, unchanged)
```

**2. Facets was a `DISTINCT` over a million rows to find 37 pairs.** Replaced with a
recursive skip-scan over `idx_rsr_order`'s `(payer, network)` prefix — Postgres has no loose
index scan, so this walks 37 index probes instead of a Parallel Seq Scan + HashAggregate:

```
SELECT DISTINCT payer, network   143 ms  ->  22 ms   (37 rows, unchanged)
```

Band-matview cost of going 5 → 20 codes, for the record (this is why no work was needed
there): `rate_bands_license_summary` **0.607 ms → 1.409 ms** (468 → 1,314 rows);
`rate_bands_payer_summary` 65 → 184 rows, both under 2 ms.

## Flags for the lead

1. **The cap change is the one judgment call here.** 100 → 400 preserves every currently
   visible row but also admits **+386 groups** that /rates excluded before, so the Services
   corpus grows 452,168 → 1,091,544 service rows. That is a deliberate, measured widening,
   not a side effect — but it does change what the Services tab lists, so it should be your
   call to keep. Reverting is a one-line change to `HAVING count(*) <= 400` plus a rebuild.

2. **698 MB for one matview.** `rate_service_rows_mv` is the largest object in the rates
   layer. It refreshes CONCURRENTLY in 26 s so it is operationally fine, but it is worth
   knowing before the corpus grows again.

3. **`ops/harvest/sync-plan.mjs` is ops-agent's file.** I added one line to register the new
   matview — a matview nothing refreshes goes stale silently, so leaving it out was not an
   option. Staged as its own hunk; flagging the seam crossing.

4. **`/published-rates` and `/recruiting` still show five codes.** Both read `RATE_CODES`
   (`lib/rate-table.ts`) and the 5-code pivot in `sql/032`/`sql/027`, which I deliberately
   left untouched. They were outside this tranche's ask; if they should also carry 20, that
   is a second migration on the parent tree and wants its own brief.

5. **Fewer insurers on the newer codes is real data, not a bug** — see the honesty note in
   premise correction 1. If a surface ever wants to explain the thinner coverage, the payer
   counts are in `rate_bands_payer_summary`.
