## What shipped (`756bdcc`, pushed in `b04b891`)

`.github/workflows/nightly-rebuild.yml` ‚Äî a scheduled GitHub Actions workflow
that is the cloud fallback for nights the Mac runner didn't fire.

- **Same chain, same code path, zero fork.** The workflow calls a new thin
  entry `ops/harvest/rebuild-daily.mjs`, which calls the exact
  `runDailyRebuild()` the Mac runner runs as its last phase ‚Üí the shared
  `ops/harvest/sync-plan.mjs` (10 `REFRESH CONCURRENTLY` + 4 `ANALYZE`). The
  belt cannot diverge from the suspenders ‚Äî same views, same order, same sound
  skip-guard, same `sync_runs` ledger. The entry is dual-use: `node
  ops/harvest/rebuild-daily.mjs` locally too.
- **It's a no-op on normal nights.** The skip-guard ("rebuild only if something
  loaded since the last ok daily") means that when the Mac already rebuilt, the
  belt sees the fresh `daily` row and skips ‚Äî it does real work *only* when
  nothing rebuilt since the last load. So belt and suspenders never double-work.
- **Legible ledger.** Belt rows write as `daily | belt` (via `SYNC_TRIGGER`,
  env-gated in `openRun` ‚Äî the Mac stays `daily | cron`). Because the belt only
  writes a row when it *rebuilds*, any `daily | belt` row is instantly readable
  as "the cloud caught a night the laptop missed." The `/insights` card treats
  it exactly like any daily run.
- **Schedule:** `13 11 * * *` UTC (06:13 EST / 07:13 EDT) ‚Äî comfortably after
  the Mac's 01:04 load+rebuild window in both DST states, off-minute (GitHub
  cron is best-effort; can run 10-30m late or drop under load ‚Äî fine for a belt).
- **Hygiene:** `permissions: contents: read` (least privilege), a `concurrency`
  guard (no overlapping rebuilds), `timeout-minutes: 25`, and a guard step that
  makes an unconfigured run a **clean green no-op** (not a red failure) so
  there's no failure-email noise before the secret is added.
