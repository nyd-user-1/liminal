## What shipped

`.github/workflows/nightly-rebuild.yml` ‚Äî a scheduled GitHub Actions workflow,
the cloud fallback for nights the Mac runner didn't fire.

- **Same chain, same code path, zero fork.** The workflow calls a new thin entry
  `ops/harvest/rebuild-daily.mjs` ‚Üí the exact `runDailyRebuild()` the Mac runner
  runs ‚Üí the shared `ops/harvest/sync-plan.mjs`. When another session added an
  11th matview (`org_network_rates`, sql/048) to that shared plan mid-session,
  the belt picked it up automatically ‚Äî which is the entire point of the shared
  plan. The entry is dual-use: `node ops/harvest/rebuild-daily.mjs` locally too.
- **No-op on normal nights.** Same sound skip-guard: when the Mac already
  rebuilt, the belt sees the fresh `daily` row and skips; it does real work only
  when nothing rebuilt since the last load.
- **Ledger.** Belt rebuilds ledger as `daily | cron` ‚Äî the same as the Mac's.
  (See the ledger note below: `sync_runs.trigger` is CHECK-constrained to
  `cron`/`manual`, and a scheduled belt *is* a cron.) A belt rebuild is told
  apart from a Mac one by its run time (~11:13 UTC, well after the Mac window),
  not a distinct label.
- **`force` dispatch input** ‚Äî `workflow_dispatch` with `force=true` sets
  `DAILY_FORCE=1`, bypassing the skip-guard. An operator lever (mirrors the
  runner's `DAILY_FORCE`) for an out-of-band load the ledger didn't see ‚Äî and
  how the psql-rebuild path was exercised end-to-end below.
- **Schedule:** `13 11 * * *` UTC (06:13 EST / 07:13 EDT) ‚Äî after the Mac window
  in both DST states, off-minute (GitHub cron is best-effort; fine for a belt).
- **Hygiene:** `permissions: contents: read`, a `concurrency` guard,
  `timeout-minutes: 25`, and a guard step that makes an unconfigured run a clean
  green no-op (no failure-email noise).

## The ledger bug this verification caught (and the fix)

The forced run rebuilt fine but wrote **no `sync_runs` row**. Root cause: I had
tagged belt rows with `SYNC_TRIGGER=belt` for legibility ‚Äî but
`sync_runs.trigger` has a CHECK constraint `trigger IN ('cron','manual')`, so
`openRun`'s INSERT threw, was caught ("the ledger is a convenience"), and no row
was written. The rebuild ran; the ledger silently didn't. I'd assumed the column
was free-form; it isn't.

**Fix ‚Äî dropped the `belt` label rather than widen the schema.** A permanent
CHECK-constraint migration for a cosmetic ledger label isn't worth it: `cron` is
honest (a scheduled belt *is* a cron), and the run timestamp already
distinguishes a belt rebuild (~11:13 UTC) from a Mac one (~07:00‚Äì09:00 UTC).
Reverted `openRun` to `'cron'` and removed `SYNC_TRIGGER` from the workflow.

**Why no re-run was needed to confirm the fixed ledgering** (proof by
composition of already-verified facts, not a redundant 340s rebuild):
1. the neon HTTP client works from GitHub ‚Äî run 29663135223's skip-guard SELECT
   succeeded;
2. `openRun` INSERT with `trigger='cron'` works ‚Äî 12 such rows exist, including
   my own 17:08 forced rebuild through this exact path;
3. the CHECK allows `'cron'`.
Same transport for SELECT and INSERT, a write-capable role ‚Äî so the belt's
`openRun('cron')` from GitHub writes its row. The only thing that had blocked it
was the `'belt'` value, now gone.
