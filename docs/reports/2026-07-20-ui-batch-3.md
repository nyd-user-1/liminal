## 5. Portal search — decision and evidence

### The decision: a separate endpoint

Built `app/api/portal/search/route.ts` rather than widening `/api/search`.

`/api/search` is `requireRole("practitioner")` and fans out over four corpora: `listClients` (the entire roster, unfiltered), `searchProviders`, `listOrgs`, `searchEmployers`. Three of those must never reach a patient and the fourth is every other client. Widening it means four conditionals inside one handler where the difference between correct behaviour and a PHI disclosure is one `if` somebody forgets during a later edit. That is the wrong shape for this boundary, so the portal got its own route and the shared one kept its single audience.

**The scoping is structural, not a filter.** The client id comes from `clientForUser(session.user.id)` — the same `clients.user_id` mapping every portal page uses — and the route never reads an id from the request. There is no parameter to tamper with because there is no parameter. Every repo call is constrained by that id at the SQL level; nothing is fetched broadly and narrowed afterwards, so no other client's row enters the process at all.

Care team is narrowed to `client.primaryPractitionerId` before any matching, so the practitioner roster is never exposed.

### Audit

`logEvent` fires on every executed search: actor, `entity: "client"`, `entityId` = the searcher's own client id, and per-group result counts. Confirmed writing to `audit_events` (6 rows during testing, `entity_id` always Casey's own id — including under every tampering attempt).

The raw query is deliberately **not** logged. A search term is free text of unknown sensitivity ("HIV", a medication name) and `audit_events` is append-only and widely read; the *access* is what has to be provable and the term adds nothing to that. Reversible call — flag it if an auditor wants the terms.

### Negative testing

Run against live data as `casey@liminal.demo`, using a real second client (Jordan Lee, `…002002`, invoice `INV-2026-0001`) and other clients' real file names, note titles and thread subjects.

**Cross-client terms — all 200, all zero hits:**

| query | matches (another client's) | result |
|---|---|---|
| `Lee` | surname | `{"groups":[]}` |
| `Jordan` | given name | `{"groups":[]}` |
| `jordan.lee@example.com` | email | `{"groups":[]}` |
| `INV-2026-0001` | invoice number | `{"groups":[]}` |
| `prior-records-dr-feld` | file | `{"groups":[]}` |
| `med check` | note title | `{"groups":[]}` |
| `Intake paperwork reminder` | thread subject | `{"groups":[]}` |
| `Superbill for June` | thread subject | `{"groups":[]}` |
| `Ava`, `Parker` | other clients | `{"groups":[]}` |

The database holds 13 invoices; Casey's `q=INV` returns exactly her 4.

**Parameter tampering — a real other client's id, eight parameter names.** Every one returned 200 with the *searcher's own* four invoices and never the named client's:

```
200  hits=4  clientId=<Jordan Lee>   ["INV-2026-9003","INV-2026-9002","INV-2026-9001","INV-2026-0004"]
200  hits=4  client_id=…  clientid=…  id=…  patientId=…  as=…  scope=…  userId=…   (identical)
200  hits=4  q=INV&clientId=<Jordan>&q=Lee                                          (identical)
200  hits=4  spoofed x-client-id / x-user-id / x-forwarded-user headers              (identical)
```

Ignored, as designed — the route never reads them.

**Corpus isolation:** `q=psychiatry` → `{"groups":[]}`, `q=headway` → `{"groups":[]}`. The portal route has no directory, rates, org or plan corpus to reach.

**Role boundaries:**

```
401  {"error":"Sign in required."}                              anonymous  → /api/portal/search
403  {"error":"This action requires the client role."}           practitioner → /api/portal/search
403  {"error":"This action requires the practitioner role."}     client → /api/search
```

`requireRole("client")` does not admit admins the way `"practitioner"` does, so the admin login cannot reach a patient's portal search either.

**In the UI:** typing `INV` lists only Casey's four invoices (`08`); typing `Jordan` shows "No matches." (`09`).
