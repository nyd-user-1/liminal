## What shipped

### NYS-129 (PRIMARY) — the harvestd runner *is* the nightly matview rebuild

The runner now rebuilds the matviews itself, via **psql**, as the **last phase
of the night** — after every load. It runs the exact ten `REFRESH MATERIALIZED
VIEW CONCURRENTLY` + four `ANALYZE` chain, in the exact dependency order, and
ledgers it as job `daily` with `timeout_ms`, riding the existing email + bell +
exit-code plumbing. Two structural choices:

- **Post-loads phase in the 01:04 runner, not a second 05:45 LaunchAgent.**
  Putting the rebuild inside the runner makes the loads→rebuild ordering a *code
  fact* (sequential, same process, after the last load) instead of a schedule
  coincidence between two agents — the ordering memory calls load-bearing. One
  fleet to keep alive, one lid-physics exposure, not two.
- **psql, not the neon HTTP driver.** A real Postgres session has no 300s
  ceiling (undici's `headersTimeout`, the NYS-65 family); the HTTP driver does.
  This is the path the lead's proven 6m07s manual rebuild took. The password
  never lands in `argv` — the runner parses `DATABASE_URL` into `PG*` env vars
  so `ps` can't see it.

**No-fork guarantee.** VIEWS / ANALYZE_TABLES / IDENT moved into one shared
module, `ops/harvest/sync-plan.mjs`, imported by *both* the runner (plain node
ESM) and `app/api/cron/daily/route.ts` (allowJs + bundler resolution). Add a
matview once; both executors pick it up. The insidious failure this kills: a
future sql/0XX matview added to the route but not the runner (or vice-versa)
would silently go stale in the primary path.

**The skip-guard — corrected from the brief's literal wording.** "Skip if a
`daily` row already succeeded today" is *unsound*: an off-cadence rebuild (an
operator's manual one, the demoted Vercel route) landing within the day window
of the nightly would make the nightly skip **even though tonight's loads still
need reflecting** — staleness, the worst failure this system exists to prevent.
Implemented the sound version instead: **skip only if nothing has loaded since
the last successful rebuild** (the newest ok `daily` is newer than every other
`sync_runs` row) *and* it's recent (the recency bound still forces a periodic
rebuild for fresh planner stats on an idle stretch). The instant tonight's jobs
run, they out-date the last rebuild and the guard correctly declines to skip.

**Verified live.** A forced full rebuild through the runner's own code path left
a genuine `daily | cron | ok` row: **314,576 ms, 14 steps** (10 views + 4
analyze), `timeout_ms` 1,500,000. Per-view timings ledgered. psql-over-pooler
refreshed all ten with no ceiling. The corrected guard then skipped a second
call citing *"nothing loaded since the ok rebuild 6m ago."* `/insights → Nightly
sync` renders **Healthy** (headless login + fetch, both 200) off that row.

### NYS-132 — run-payer.sh stops reporting empty harvests as success

The Emblem false-success root cause: per-file `curl`/`scan` failures didn't fail
the job, and the load step happily loaded empty CSVs, so the runner logged "ok."
Fixed at the source:

- **PIPESTATUS** — any nonzero stage of `curl | decomp | scan` fails that file.
- **`curl -f`** — an HTTP 4xx/5xx is now a failure, not an error body piped on.
- **rows > 0** — scan-tic always writes a header line, so a real harvest is ≥ 2
  lines; a header-only (or missing) CSV = zero providers matched = failure.
- **legitimate-zero override** — an optional 7th manifest field (`…|zerook`)
  allows empty output for a known provider-less network (e.g. Emblem's HCP).

Any file failure → nonzero exit → the `&& load` short-circuits → the runner
ledgers `error`, retries, and alerts, instead of a green tick over partial data.
Composes with NYS-124: PIPESTATUS/rows>0 is the source-level honesty (exit
nonzero), the suspect-guard is the timing backstop.

**Verified** — 5/5 cases: happy (exit 0, ROWS=30) · bad-gz (`PIPESTATUS 0 1 4`
→ fail) · curl-fail on a missing file (`37` → fail) · 0-rows no-override (fail)
· 0-rows `+zerook` (exit 0, ZERO-OK).

### NYS-130 — ruling on the demoted Vercel route: **manual-only**

Removed the `crons` entry from `vercel.json`; the route no longer auto-fires. It
survives as an authenticated manual/emergency trigger (Bearer `CRON_SECRET`,
still 401s without it — verified). Recorded in the route header and the runbook.

**Why manual-only, not split-to-fit.** The runner is now the reliable automatic
path. The Vercel route failed in *both* dimensions the belt was supposed to
cover: Hobby delivery is best-effort and silently skipped its window (the
2026-07-18 incident), and — even when the founder manually delivered it that same
day — it was guillotined at exactly `300000 ms`. Tonight's live rebuild measured
**315s** for the full chain (org_tin_rosters alone is now 108s, up from a
measured 8.8s, on the 13.4M-row corpus), so the chain no longer fits one
function under any circumstances. A belt that both skips *and* times out isn't
worth the app-surface of a split. Split-to-fit was considered and declined; its
only benefit is an automatic *cloud* belt, and that belt is doubly-broken.
