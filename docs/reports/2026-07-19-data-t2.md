**Three separable killers, each now measured:**

1. **Single-pass refs retention — the historical 6 GB OOM.** The Jul-12
   instrumented run retains ~24% of ALL provider groups (37,376 of 155,871
   refs by 581 MB; 379,813 by 5.9 GB — heap tracking it 456 MB → 4.3 GB,

   single-pass retention ≈ 3.5M groups ≈ tens of GB. No heap cap can work —
   the JSON.parse in the Jul-12 stack was the straw, not the cause. Two-pass
   retention on the same file: **2,963 groups (~3 MB)**.

   failure (most empire2/highmark2 PIPESTATUS lines are `56 1 4`). 39F0-1
   streams ~50 min; from disk, pass A crossed all 67.9 GB in **1.96 min at
   16 MB heap**. Fixed: `run-two-pass.sh` now downloads once (resumable,
   `--retry 20`), scans both passes from disk, deletes after.
3. **Pass-A default-heap OOM (exit 134, 062-series).** Matched items near the
   200 MB `HUGE_ITEM` line JSON.parse into multi-GB transient trees; pass A
   ran unsized. Fixed: pass A gets the same `--max-old-space-size=6144`.

**Parsed rows proving the fix (acceptance a):** pass A collected 16,350 gids
