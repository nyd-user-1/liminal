## Flags for the founder

1. **PHI leak fixed, pre-existing.** `/api/stripe/checkout` was sending every
   invoice line description to Stripe (`product_data.description`) — that field
   is where service names live ("Psychiatric diagnostic evaluation"). Stripe
   signs no BAA. Line items are now `INV-… — Therapy session`. This route
   predates tonight; the leak has been live in the mock path and would have gone
   live with the first real key. Worth checking whether any other outbound
   integration does the same.

2. **MCC — I did not use the brief's 8049.** 8049 is "Podiatrists and
   Chiropodists". I used **8099** "Medical Services and Health Practitioners
   (Not Elsewhere Classified)", which is the standard code for non-physician
   licensed practitioners (LCSW/LMHC/psychologist). MCC drives card-network
   categorization and some issuers' HSA/FSA eligibility logic, so it is a claim
   about what the business *is*. A psychiatrist (MD) billing under their own
   account would be 8011 — revisit when we onboard one.

3. **`stripe` SDK was already at v22**, not the v20 the brief assumed (the
   sample's version). `npm install` moved it 22.2.0 → 22.3.2, inside the
   existing `^22.2.0` range, so the lockfile would have resolved there anyway.
   Existing invoice code typechecks clean against it.

4. **Webhook now returns 200 on handler failure**, with the reason recorded in
   `stripe_events.error` and `processed_at` left NULL. Stripe's retry storm
   doesn't help when the bug is ours; the unfinished row is the durable signal.
   `SELECT * FROM stripe_events WHERE processed_at IS NULL` is the worklist —
   somebody should watch it once real events flow.
