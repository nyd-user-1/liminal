## What shipped

`/settings/payments` — a fourth Settings tab labelled **Get paid**. Four states
off `GET /api/connect/status`:

- **no account** — EmptyState + "Set up payments" → `POST /api/connect/account`

- **onboarding** — embedded `ConnectAccountOnboarding`, with "Open on Stripe
  instead" (Account Link) as the fallback path

- **pending** — `detailsSubmitted` but not `chargesEnabled`: Stripe's
  notification banner, outstanding requirements as Tags, account management

- **active** — status badges + account id, notification banner, and a
  Balance / Payouts / Account segmented control over the three embedded
  components, plus an Express Dashboard link

Four files, all under `app/(app)/settings/payments/`: `page.tsx` (server, thin),
`payments-client.tsx` (state machine), `connect-embed.tsx` (runtime + theming),
`connect-api.ts` (route contract).

**Kit primitives only** — SettingsCard, Banner, EmptyState, SegmentedControl,
Badge, Tag, Button, Spinner, Card. No new primitives, no feature-component
duplication. The Stripe iframes are themed to the light-mode tokens (teal
`#3f8290`, navy ink, 12px/8px radii, Inter) so they read as part of the app;
the authenticated shell never renders dark, so a static map is correct.

Canonical layout rules respected: no page-level H1 — TopBar owns "Settings",
and the Refresh action goes through `TopBarActions`.

## Seam discipline

The UI never imports `lib/repos/stripe-connect.ts`. Every transition goes
through five calls in `connect-api.ts`, so this seam only had to agree with
Terminal A on JSON. When their routes landed mid-session the contract matched
on the first try (`{ account: … }`, `{ clientSecret }`, `{ url }`, camelCase).
The normalizer also accepts snake_case, since a route passing a Stripe object
straight through is a live possibility.

Untouched, as instructed: `app/api/**`, `lib/email/`, portal pages,
`components/rates/**`, `app/(app)/workspace/**`.

## The CSP — I did not do what the brief asked, deliberately

The brief said "CSP updated for Stripe frames/scripts". **The app had no CSP at
all**, so nothing was blocking Stripe to begin with; the stated goal was already
satisfied by absence.

Adding an *enforcing* policy is a much larger change than it reads. It has to
cover Photon's e-prescribing runtime (`api`/`app`/`clinical-api` across the
photon, neutron and boson domains, plus Auth0), Vercel Blob, and Next's own
inline bootstrap scripts — which need either `'unsafe-inline'` or a nonce from
middleware this app doesn't have. Every wrong guess fails silently in the
browser, and e-prescribing is the surface that breaks.

So it ships as `Content-Security-Policy-Report-Only`: the Stripe directives are
written down and reviewed now, violations log, nothing blocks. The header
comment lists the three steps to enforce it. **This is an open item, not a
finished one** — flagging it rather than closing it under a green check.

## Flags for the lead

1. **BLOCKER — no Stripe keys exist.** `.env.local` has no
   `STRIPE_SECRET_KEY`, no `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY`, no
   `STRIPE_WEBHOOK_SECRET`. The founder checklist assumed the existing invoice
   checkout implied they were there; it doesn't — `lib/stripe.ts` degrades to a
   mock when the key is absent, which is why nobody noticed. T6 cannot pass and
   three of my four states cannot be driven until the founder does the Dashboard
   step. My surface degrades honestly meanwhile (clear warning, no crash), and
   that path is the one state I verified for real.

2. **Route-path collision, resolved by moving my page.** Terminal A's
   account-link return/refresh URLs and both payout email CTAs pointed at
   `/settings/payments` while my surface shipped at `/settings/get-paid` — three
   files across two seams, all 404 in practice. I moved the page rather than
   edit routes and templates I don't own. Tab label stays "Get paid". Worth a
   ruling on the canonical name so it doesn't drift back.

3. **`POST /api/connect/login-link` does not exist.** The T2 route list omits
   it, but T3 calls for an Express Dashboard link. The button is wired and
   degrades to a toast if the route 404s. Either Terminal A adds it or we drop
   the button — the embedded account-management component already covers most
   of what the dashboard does.

4. **Concurrent-tree incident.** My route move (`27a57e1`) was committed by the
   T4/T5 session, which ran `git commit` while my files sat staged in the shared
   index. Content is intact and correct in HEAD, but the history now attributes
   a settings route move to a docs commit. Staging-only discipline isn't
   sufficient protection when another session commits without pathspecs.

5. **No fee percentage in the UI.** The 10% application fee is an unresolved
   business decision (open fork #1). I deliberately did not print a placeholder
   rate as if it were policy — the copy says "minus a platform fee" and stops.
   Needs real copy once pricing is decided.
